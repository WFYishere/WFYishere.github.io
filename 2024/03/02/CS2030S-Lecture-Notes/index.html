    Java

    1. \(Java\) \(Compiler\): compiled into bytecode.(.java-> javac -> .class); interpreted and compiled on the fly by JVM into machine code`.(java .class) / be interpreted by the Java interpreter.

    2. \(Types\):

    • \(statically\) \(typed\) \(language\): declare every variable we use in the program and specify its type(compile-time type).

      • \(Primitive\) Types: numeric & boolean values byte <: short <: int <: long <: float <: double; char <: int

      • \(Reference\) Types: stores only the reference to the value

    • widening type conversion: long i = int(j);

      narrowing type conversion: type casting

    1. Principals:

    • \(Encapsulation\): keep abstraction barrier.

    • \("Tell,\) \(Don't\) \(Ask"\) principle: E.g. The client should tell a Circle object what to do , instead of asking "What is your radius?"

    • \(Composition\): combining classes

    • \(Composition\) : has-a relationship; \(Inheritance\): is-a relationship.

    • \(Polymorphism\): method invoked is decided during run-time(\(dynamic\) \(binding\)/\(dispatch\)) - Class Methods do not support dynamic binding. At compile time, determines the meth od descriptor(including return type) of the method invoked, using the compile-time type of the target. At runtime, the method descriptor is retrieved and the run-tim type is determined

    • \(Inheritance\): subtypes can only access public fields/methods

    • \(Liskov Substitution Principle\): A subclass should not break the expectations set by the superclass

    • method overloading ： differing method signature.

    1. abstract: cannot be instantiated - concrete class

    2. \(Interface\):

    • across different class hierarchies.(public abstract by default.) An interface is a type (a type can have multiple supertypes.e.g. a class can be subype of multiple interface)

    • casting unrelated interface is allowed. E.g.: class A, Interface B, class C extends A implements B. There is possiblity that C(subclass of A) can be casted on B. In this situation A = B compiles. default implementation: default (impure interfaces)

    1. \(Wrapper\) \(Classes\) :

      primitive wrapper class objects are immutable

      • \(covariant\) : S<:T -> C(S) <: C(T); \(contravariant\);\(invariant\): neither Arrays of primitive types are invariant

    3. Exceptions

    • instances that are a subtype of Exception.

    • \(Checked\) Exceptions: programmer's errors. Would not happen if perfect code was written;

    • \(Unchecked\) Exceptions:InvalidInputException, subclass of RuntimeException

    1. only reference types can be used as type arguments.

    • Note that we use extends for interface bounding

    • the constructor for a generic class Pair<T> is still declared as Pair (without the type parameters).

    • Once a generic type is instantiated, it is called a parameterized type

    1. \(Type\) \(erasure\): after type checking, erase the type parameters and type arguments during compilation(to Object). If the bounded, replaced by the bounds. After type erasure, we do explicit typecast on an object.

    • arrays are \(reifiable\) type(gets full type information during run-time); but generics are not due to type erasure

    • Actually, generic array declaration is fine but generic array instantiation is not.

    • public void copyTo(Seq<? super T> dest) super is a contravariance type

    • PECS: Producer Extends Consumer Super

    • Seq<Object> is NOT supertype of all Seq<T> due to generics being invariant.

    • Seq<?> is a sequence of objects of some specific, but unknown type;

    • Seq<Object> is a sequence of Object instances, with type checking by the compiler;(Note that generics are invariants so Seq<Object> !>: Seq<Integer>)

    • Seq is a sequence of Object instanc es, without type checking.

    1. \(Immutability\):

    • the instance cannot have any visible changes outside its abstraction barrier.

    • To make a method in a class immutable, it is recommended to declare the class as immutable final too as subclasses can override the method([S]why not declare the method as final? - the final keyword prevents assigning new values to the field, but if the field is referencing a mutable object, it is possible to mutate it. So we also need to ensure that the fields themselves are immutable.)

    • advatanges: allow sharing of reference between objects.

    1. \(Nested\) \(Classes\)

    • Inner class: non-static nested class

    • When we call this.x in a nested class B in class A, this refers to instance field in B rather than A. To call the x field in A, we use A.this.x. This is called a \(qualified\) this.

    • if a nested class is private, we cannot access an instance of nested class outside the class. However, for nested class B in class A, if there is a method in A B getB() {return new B();}, we can return an instance of B outside A. So we can do: Object b = a.getB();, but we cannot do A.B b = a.getB()

    • However, as long as the nested class B does not expose any detail of class A, it is enouraged to declare B as a public type. Then we can use A.B b = a.new B();

    • \(local\) \(class\): non-static nested class in a method. A local class makes a copy of local variables inside itself. This is called \(capturing\).

    • Java only allows local classes to access variables that are effectively final. They cannot be reassigned after initializstion. However, if the variables are reference type, they cannot be reassigned but can be mutated. See the below example. The \(capturing\) is done after the instantiation of a nested class - the list is sorted in descendingOrder

    1. \(Pure\) \(Functions\): deterministic(return same y for same x), referential transparency(same x for same y), no side effects(no printing/exception/changing of other variables etc.).

    • \(Functional\) \(Interface\): interface that has a single abstract method

    • \(lambda\): Transformer<Integer, Integer> t = (/*Integer*/ i) -> (i + 1);, where Transformer is a functional Interface. A lambda function is the same as a local class so it captures local variables used. Thus we store the function together ith the enclosing environment - \(closure\).

    • \(Function\) \(Reference\): Transformer<Point, Double> t = origin::distanceTo(); which is the same as: t = p -> origin.distanceTo(p)

    • The technique of translating a general n-ary function to a sequence of n unary functions is called \(currying\). e.g. instead of (x,y) -> x+y, we do x -> y -> x + y. These functions are called \(curried\) \(fcuntions\). With currying, we can partially apply a function. This is useful if one of the arguments doen't change often/ is too expensive to compute.

    1. \(Lazy Evaluation\):

    • \(Procrastinate\): when we create a lambda, its content is not evaluated. Only when it is called the content is evaluated.

    • \(Memoization\): Only works when the function is pure!

    1. \(Stream\)
    • Limitation: stream can only be operated once. We cannot use terminal operation on a stream multiple times. We cannot iterate a stream multiple times. This would throw IllegalStateException
    1. \(Monad\):

    • The \(left\) \(identity\) \(law\):Monad.of(x).flatMap(x -> f(x)) must be the same as f(x)

    • The \(right\) \(identity\) \(law\): monad.flatMap(x -> Monad.of(x)) must be the same as monad

    • \(associative\) \(law\): monad.flatMap(x -> f(x)).flatMap(x -> g(x)) must be the same as monad.flatMap(x -> f(x).flatMap(y -> g(y)))

    1. \(Functor\)

    • preserving identity: functor.map(x -> x) is the same as functor

    • preserving composition: functor.map(x -> f(x)).map(x -> g(x)) is the same as functor.map(x -> g(f(x)).

    1. \(Parallel\) \(Computing\)

    • s.parallel().filter(x -> x < 0).sequential().forEach(..); If we call parallel and sequential in the same stream, the last call wins.

    • Conditions of parallelable

      • \(Interference\): one of the operations modify the source stream when the terminal operation is executed.

      • \(Stateful\): the lambda expression's result depends on a state that might change

    • Parallelising Reduce:

      • combiner.apply(identity, i) = i; combiner and accumulator must both be associative and compatible
    1. Side Notes:
    • T... items are called varargs. The methods that use \(varargs\) are called \(variadic method\). Use @SafeVarargs to tell te compiler that we know what we are doing.
